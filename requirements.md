# AI 智能起名系统需求说明

## 1. 项目目标

本项目面向需要中文起名、企业品牌命名和宠物命名的用户，提供基于 AI 的候选名字生成、多轮反馈优化、项目化管理、历史收藏、报告导出、会员额度充值和企业私有知识库增强能力。

系统应支持普通用户完成注册登录、起名生成、历史管理、会员额度使用和支付宝沙箱充值；支持管理员维护用户、会员套餐、订单、知识库任务和核心业务数据。

## 2. 用户角色

- 普通用户：注册登录、配置偏好、创建命名项目、生成名字、查看历史、收藏名字、导出报告、上传企业知识库文件。
- 会员用户：在普通用户能力基础上，通过支付宝沙箱充值或管理员授予获得更多起名额度。
- 管理员：查看系统数据、维护用户和套餐、调整额度、处理订单退款、管理知识库解析任务、维护白名单业务数据。

## 3. 功能需求

### 3.1 注册登录

- 用户可以通过邮箱验证码注册账号。
- 用户可以使用邮箱和密码登录。
- 用户可以重置密码、修改个人资料、修改密码和邮箱。
- 系统记录登录成功和失败记录。
- 管理员接口必须校验管理员身份。

### 3.2 AI 起名

- 支持 `人名`、`企业名`、`宠物名` 三类起名。
- 人名起名应支持姓氏、性别气质、字数要求、期望寓意等输入。
- 企业名和宠物名应支持行业、风格、地区、字数、补充说明和排除词等输入。
- 起名接口应返回候选名字、寓意解释、拼音或英文名等结构化信息。
- 起名生成应消耗会员额度，失败时退回本次消耗额度。

### 3.3 多轮反馈

- 首次起名生成后返回 `thread_id`。
- 用户可以基于已有 `thread_id` 提交反馈继续生成。
- 多轮反馈应参考历史上下文和用户新增要求。
- 多轮反馈也应消耗额度，失败时退回额度。

### 3.4 项目管理

- 用户可以创建、查看、编辑和归档命名项目。
- 起名记录可以关联到项目。
- 项目详情应汇总关联的候选名字和历史结果。
- 项目支持导出命名报告。

### 3.5 偏好模板

- 用户可以创建、查看、编辑和删除偏好模板。
- 模板应支持按起名分类保存常用偏好。
- 用户可以设置默认模板。

### 3.6 历史与收藏

- 每个候选名字应保存为一条起名历史。
- 用户可以查看历史列表和历史详情。
- 用户可以收藏或取消收藏历史名字。
- 用户可以对多个历史名字进行对比。
- 历史记录支持导出报告。

### 3.7 名字评分

- 系统应对候选名字给出综合评分。
- 评分维度包括音律、寓意、传播、域名等适用指标。
- 评分结果应保存到历史记录中。

### 3.8 企业知识库 RAG

- 企业名场景支持上传 TXT/PDF 知识库文件。
- 上传后创建异步解析任务，任务状态包括 `queued`、`processing`、`done`、`failed`。
- Worker 解析文件并写入 Chroma 向量库。
- 企业名生成时应检索当前用户启用的私有知识库内容。
- 用户和管理员可以查看任务状态并触发重新解析。

### 3.9 会员额度与支付

- 新用户首次创建额度账户时默认获得免费额度。
- 用户可以查看套餐、当前会员额度和额度流水。
- 用户可以创建充值订单。
- 充值订单应生成支付宝 `page` 或 `wap` 支付链接。
- 支付宝异步通知应完成验签，并根据交易状态更新订单和额度。
- 用户应能主动查询支付宝订单状态，同步更新本地订单状态。
- 历史 mock 支付接口应保持关闭，避免绕过支付流程直接增加额度。
- 管理员可以创建或更新套餐、调整用户额度、重置用户密码和退款订单。

### 3.10 报告导出

- 历史记录和项目支持导出报告。
- 支持格式包括 `pdf`、`image`、`txt`。
- 导出内容应包含候选名字、解释、评分和相关输入信息。

### 3.11 管理员后台

- 管理员可以查看用户、起名历史、生成统计、知识库任务统计和知识库任务列表。
- 管理员可以触发单个或批量知识库任务重新解析。
- 管理员可以通过白名单数据维护接口管理指定业务表。
- 管理员数据维护不得暴露或直接修改用户密码哈希。
- 禁止管理员删除当前登录管理员账号。

## 4. 非功能需求

- 安全性：所有需要登录的接口使用 JWT Bearer Token；管理员接口使用管理员鉴权；真实密钥、支付私钥不得写入源码、文档和日志。
- 可维护性：后端按路由、模型、schema、repository、core 分层；前端以 `uniapp-demo01/pages/`、`http/`、`utils/` 为主源码。
- 可靠性：起名失败、外部服务异常和知识库解析失败应返回明确错误，并尽量保持额度和任务状态一致。
- 可测试性：自动化测试应 mock LLM、RabbitMQ 等外部依赖，使用临时数据库验证核心业务规则。
- 兼容性：前端支持 HBuilderX/uni-app 调试；本机 H5 默认访问 `http://127.0.0.1:8000`，真机或模拟器需配置电脑局域网 IP。
- 数据持久化：MySQL、PostgreSQL、上传目录和 Chroma 目录在部署时应使用持久化存储。

## 5. 验收标准

- 用户可以完成注册、登录、密码重置和个人资料维护。
- 用户可以在三种分类下生成名字，并在额度不足时收到明确提示。
- 起名生成成功后历史记录落库，生成失败后额度退回。
- 用户可以基于 `thread_id` 继续反馈生成。
- 用户可以创建项目、关联起名记录、查看项目详情并导出报告。
- 用户可以创建偏好模板并在后续起名中复用。
- 用户可以查看历史、收藏名字、对比名字并导出历史报告。
- 企业名用户可以上传知识库文件，Worker 解析成功后企业名生成能使用 RAG 上下文。
- 用户可以查看会员套餐、充值订单、额度流水和当前额度。
- 用户可以创建支付宝沙箱充值订单，获得支付链接，并通过异步通知或主动查询完成订单状态同步和额度增加。
- mock 支付接口不可用于直接完成支付。
- 管理员可以访问后台接口，普通用户访问管理员接口会被拒绝。
- 管理员数据维护接口只允许白名单表和允许字段。
- 自动化测试通过，后端 Python 编译检查通过。

## 6. 暂不支持或后续规划

- 支付宝当前面向沙箱联调，正式收款前仍需完成生产应用配置、证书/密钥管理、公网回调、对账和退款链路校验。
- 微信支付尚未完整接入，当前保留配置项作为后续扩展。
- 生产级监控、告警、日志聚合和备份恢复策略需要在正式部署前补充。
- 小程序、App Store 或应用市场发布流程尚未形成标准文档。
- 多租户企业管理、团队协作和企业级权限模型可作为后续扩展。
